# 07 - 风险与约束

## 1. 风险矩阵

| 风险 | 等级 | 影响 | 应对措施 |
|------|------|------|----------|
| OTP fuse 后无法回退 | 高 | 变砖 | fuse 前充分测试；保留 MaskROM 救砖 |
| HVB 校验失败导致无法启动 | 中 | 设备不可用 | unlock 调试模式；rockusb 恢复 |
| 分区改表需重刷全机 | 中 | 升级成本高 | Phase 0 一次性规划 rvt，后续不改 GPT |
| dm-verity 增加启动时间 | 低 | 开机慢几百 ms | 大分区用 hashtree，只校验 root hash |
| HVB 与 AVB 双栈冲突 | 中 | 维护复杂 | 统一 bootkhp + HVB 主路径 |
| 私钥泄露 | 高 | 伪造签名 | 离线 HSM；PRK 与 PSK 分离 |
| RPMB 数据损坏 | 低 | 回滚失效 | OP-TEE RPMB FS 已有冗余 |
| OTA 未重签名 | 中 | 升级后无法启动 | CI 集成 ota_resign.sh |

---

## 2. 技术约束

### 2.1 平台约束

- **SoC**：RK3568，BootROM 仅支持 RSA 验签（非国密硬件加速）
- **存储**：eMMC（fe310000.sdhci），RPMB 分区可用
- **TEE**：OP-TEE v2，支持 OTP / RPMB / U-Boot store TA

### 2.2 OpenHarmony 约束

- HVB 官方文档标注：**仅支持标准系统设备**（KaihongOS L2 满足）
- HVB 1.0 当前支持 SHA256 hash / hashtree，RSA 签名
- SM2/SM3 在 libhvb 中有代码，RK3568 路径以 RSA 为主

### 2.3 分区约束

- rvt 分区建议 ≥ 512KB
- GPT 分区表改动需全量重刷
- userdata 已有 fscrypt，与 HVB 无冲突

## 3. v1.0 交付范围（最终确认）

### ✅ 包含

| 能力 | 阶段 |
|------|------|
| HVB 全分区验签 | Phase 0 |
| RVT 分区 + 签名流水线 | Phase 0 |
| dm-verity 运行时只读 | Phase 1 |
| RPMB 防回滚 | Phase 2 |
| OTP 设备锁 | Phase 2 |
| 产线 provisioning CLI | Phase 0+ |
| 串口 / CLI 状态查看 | Phase 0+ |

### ❌ 不包含

| 能力 | 原因 |
|------|------|
| BIOS 安全启动菜单 | 用户明确要求 v1.0 不做 |
| BIOS 策略配置项 | 同上 |
| BootROM fuse（Phase 3） | 需产线单独试点 |
| system 全盘加密（Phase 4） | ROI 低，按需 |
| 国密 SM2 量产 | 需硬件 + 认证评估 |

## 4. 测试要点

### 4.1 功能测试

- [ ] 正常 signed 镜像启动
- [ ] 篡改 boot_linux → 启动失败
- [ ] 篡改 system → dm-verity I/O 错误
- [ ] rollback_index 降级 → 启动失败
- [ ] unlock 态 unsigned 镜像可启动
- [ ] lock 态 unsigned 镜像不可启动
- [ ] OTA 升级 + 重签名 → 正常启动

### 4.2 安全测试

- [ ] 替换 U-Boot 为 unsigned（Phase 3 前预期可启动，Phase 3 后应失败）
- [ ] 替换 RVT 分区 → 启动失败
- [ ] 修改 RPMB rollback 值 → 应不可写（lock 态）
- [ ] unlock token 重放攻击 → 应失败

### 4.3 稳定性测试

- [ ] 连续重启 1000 次无异常
- [ ] 断电重启中验签不损坏 RPMB
- [ ] 低电量 / 高温下验签正常
