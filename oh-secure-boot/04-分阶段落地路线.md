# 04 - 分阶段落地路线

## 总览

| 阶段 | 名称 | 周期 | fuse 风险 | 可救砖 |
|------|------|------|-----------|--------|
| Phase 0 | 开发态软校验 | 2–3 周 | 无 | ✅ |
| Phase 1 | dm-verity 运行时保护 | 2 周 | 无 | ✅ |
| Phase 2 | 防回滚 + 设备锁 | 2–3 周 | 低 | ✅ |
| Phase 3 | 硬件信任根 L0/L1 | 3–4 周 | **不可逆** | ⚠️ 需 MaskROM |
| Phase 4 | 分区加密（可选） | 按需 | 中 | ✅ |

---

## Phase 0：开发态软校验

**目标**：跑通 HVB 全链路，设备仍可救砖，不 fuse OTP。

### 改动

1. **分区表**：从 `misc` 或 `factory` 划出 512KB–1MB 给 `rvt`
2. **U-Boot defconfig** 增加 HVB 相关配置（详见 [05-Phase0改动清单.md](./05-Phase0改动清单.md)）
3. **构建后签名**：镜像打包完成后调用 `hvb_sign.py sign`
4. **开发模式**：unlock 状态下 `hvb_verify()` 跳过校验（代码已有）

### 验收

- [ ] 串口可见 `[hvb] hvb verify finish in uboot, errorcode: 0x0`
- [ ] 篡改 system.img 后启动失败
- [ ] unlock 状态可正常启动调试
- [ ] rockusb 仍可救砖

## Phase 1：内核 dm-verity + 分区只读

**目标**：运行时防篡改，对标 iOS 只读系统 / Android dm-verity。

### 内核 defconfig 开启

```
CONFIG_MD=y
CONFIG_BLK_DEV_DM_BUILTIN=y
CONFIG_BLK_DEV_DM=y
CONFIG_DM_BUFIO=y
CONFIG_DM_ZERO=y
CONFIG_DM_INIT=y
CONFIG_DM_VERITY=y
CONFIG_DM_VERITY_VERIFY_ROOTHASH_SIG=y
CONFIG_DM_VERITY_FEC=y          # 可选，容错
```

### fstab 改造

```
/dev/block/platform/fe310000.sdhci/by-name/system    /usr       ext4  ro,barrier=1  wait,required,hvb
/dev/block/platform/fe310000.sdhci/by-name/vendor      /vendor    ext4  ro,barrier=1  wait,required,hvb
/dev/block/platform/fe310000.sdhci/by-name/sys-prod    /sys_prod  ext4  ro,barrier=1  wait,required,hvb
/dev/block/platform/fe310000.sdhci/by-name/chip-prod   /chip_prod ext4  ro,barrier=1  wait,required,hvb
```

### 签名策略

| 分区 | verity_type | 原因 |
|------|-------------|------|
| boot_linux, ramdisk, resource | hash | 体积小，整包校验 |
| system, vendor, sys-prod, chip-prod | hashtree | 体积大，按块校验 |

### 验收

- [ ] `dmsetup status` 可见 verity 设备
- [ ] 运行中 dd 修改 system 分区块 → I/O 错误
- [ ] 正常 OTA 升级 + 重签名后可启动

## Phase 2：防回滚 + 设备锁

**目标**：防止刷回有漏洞的旧版本。

### 已有基础

- `uboot_hvb.c` → `commit_rollback_indexes()` 写 RPMB
- `rk_hvb_ops_user.c` → OTP lock_state、permanent attributes
- OP-TEE：`CFG_RK_UBOOT_STORE_OTP`、`CFG_RPMB_FS`

### 落地步骤

1. 每个镜像 cert 配置 `rollback_index` + `rollback_location`
2. 产线首次启动 fuse permanent attributes（PRK 公钥 + product_id）
3. **lock 状态**：校验失败 → 拒绝启动，进 rockusb recovery
4. **unlock 流程**：工厂 PC 工具 → challenge-response → OP-TEE 验证 → 写 lock_state

### lock / unlock 行为

| 状态 | hvb_verify 行为 | 启动 |
|------|-----------------|------|
| lock (0) | 完整校验 + 提交 RPMB | 校验通过才启动 |
| unlock (1) | 跳过校验，打印 Warning | 任意镜像可启动 |

### 验收

- [ ] 刷入 rollback_index 更低的旧版镜像 → 启动失败
- [ ] lock 态下 unsigned 镜像无法启动
- [ ] unlock 后可用 unsigned 镜像调试
- [ ] unlock token 一次性 / 有效期可控

## Phase 3：硬件信任根 L0/L1

**目标**：BootROM 级 Secure Boot，防 U-Boot 被替换。

> ⚠️ **不可逆操作**：OTP fuse 后无法回退，必须产线规范 + MaskROM 救砖方案。

### RK3568 路径

1. 生成 RSA 密钥对 → 烧 OTP 公钥 hash
2. `MiniLoaderAll.ini` / SPL 签名（Rockchip `fit-core.sh` secure boot 流程）
3. 可选：`CONFIG_ROCKCHIP_PRELOADER_HVB_PUB_KEY` 对齐 HVB 信任根
4. 修改 `KHDVK_RK3568_A_TRUST.ini`：BL33 `SEC=1`

### 当前 Trust 配置

```ini
[BL31_OPTION] SEC=1    # ATF 已安全打包
[BL32_OPTION] SEC=1    # OP-TEE 已安全打包
[BL33_OPTION] SEC=0    # U-Boot 未安全打包 ← Phase 3 目标
```

### 验收

- [ ] 未签名 MiniLoaderAll 无法启动
- [ ] 未签名 U-Boot 无法启动
- [ ] 完整 signed 镜像链正常启动
- [ ] MaskROM + rockusb 救砖通道可用

## Phase 4：分区加密（可选，后期）

| 分区 | 方案 | 现状 |
|------|------|------|
| userdata | fscrypt | ✅ 已启用 |
| system / vendor | dm-crypt + verity | 工作量大，按需 |
| U-Boot env | OTP 加密（RK U-Boot store TA） | 基础设施已有 |

**建议**：工业 / 车载优先保证 userdata 加密 + system dm-verity 只读；system 全盘加密 ROI 较低。
