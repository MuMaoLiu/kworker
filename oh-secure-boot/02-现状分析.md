# 02 - 现状分析（khdvk_rk3568_a）

基于 OpenHarmony 代码仓 `device/board/{company}/khdvk_rk3568_a` 的实际配置分析。

## 1. 当前启动链

```
BootROM
  ↓
MiniLoaderAll（SPL，rk356x_spl_loader）
  ↓
U-Boot（uboot.img，FIT + HW Crypto）
  ↓
boot_android / bootkhp（当前主路径未走 HVB）
  ↓
Kernel（extlinux boot_linux.img）
  ↓
init 挂载 system / vendor / sys-prod / chip-prod / userdata
```

## 2. 已有能力

### 2.2 U-Boot 代码（已有 HVB 集成，未启用）

- `cmd/bootkhp.c`：`CONFIG_HVB_LIBHVB` 条件下调用 `hvb_verify()`
- `lib/hvb/uboot_hvb.c`：链式校验 RVT + 分区列表，提交 rollback 到 RPMB
- `lib/hvb/rk_hvb_ops_user.c`：OTP 永久属性、lock_state、RPMB 读写

### 2.4 内核

| 配置项 | 状态 |
|--------|------|

| `CONFIG_BLK_DEV_DM` | ❌ 未开 |

### 2.5 分区与挂载

**parameter.txt 当前分区**（无 rvt）：

```
uboot, misc, resource, boot, ramdisk, system, vendor,
sys-prod, chip-prod, recovery, factory, userdata
```

**fstab.rk3568**：

- system / vendor / sys-prod / chip-prod：`ro` 挂载
- **无 `hvb` 标志** → init 不会启用 dm-verity
- userdata：已启用 `fscrypt` 软件加密

### 2.6 打包

`khdvk_rk3568_a_package_file` 包含各镜像，但 **构建流水线未接入 `hvb_sign.py`**。

### 2.7 Trust 镜像

`KHDVK_RK3568_A_TRUST.ini`：

- BL31 / BL32：`SEC=1`（ATF / TEE 有安全打包）
- BL33（U-Boot）：`SEC=0`（U-Boot 尚未纳入硬件验签链）

## 3. 缺口汇总

| 缺口 | 影响 | 优先级 |
|------|------|--------|
| HVB U-Boot 配置未启用 | 启动时不验签 | P0 |
| 无 rvt 分区 | 无法存放根校验表 | P0 |
| 内核 dm-verity 未开 | 运行时无法防篡改 | P0 |
| fstab 无 hvb 标志 | init 不启用 verity | P0 |
| 构建未签名 | 镜像无 HVB footer | P0 |
| BootROM 级 Secure Boot 未 fuse | U-Boot 可被替换 | P1 |
| AVB 与 HVB 双栈共存 | 维护复杂、策略不清 | P1 |

## 4. OHVB 能力与 HVB 映射

| OHVB 目标能力 | 官方 HVB 对应 | RK3568 现状 |
|---------------|---------------|-------------|
| 验签协议 | libhvb + hvbtool | 代码在仓，**未启用** |
| 分区尾部签名区 | HVB Footer + Cert | 工具链已有 |
| 根校验表 | RVT 分区 | **分区表缺失** |
| 防回滚 | RPMB + rollback_index | U-Boot 代码已有 |
| 设备锁 | OTP/TEE lock_state | OP-TEE 已配置 |
| 内核运行时校验 | dm-verity + fstab hvb | **未启用** |
| 固件加密 | fscrypt（userdata） | userdata 已加密 |
