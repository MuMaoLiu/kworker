# 01 - 方案概述

## 1. 背景

OpenHarmony 6.1 在安全启动方面与 Android / iOS 存在明显差距：

| 维度 | Android | iOS | OH 6.1（现状） |
|------|---------|-----|----------------|
| 统一验签机制 | AVB 2.0 | Secure Boot 链 | 无官方统一 AVB 级机制 |
| 分区防篡改 | dm-verity | 系统只读 | 弱，依赖零散配置 |
| 防回滚 | rollback_index | 安全计数器 | 未体系化 |
| 安全启动 | 完整信任链 | 完整信任链 | 依赖 U-Boot 零散配置 |

工业 / 车载 / 金融场景对**启动完整性、防刷机、防降级**有刚性需求，OpenHarmony 需要在 RK3568 平台上建立完整的安全启动链。

## 2. 目标

实现 **OHVB（OpenHarmony Verified Boot）**，对标 Android AVB 2.0 + iOS Secure Boot：

- 安全启动链（Secure Boot Chain）：BootROM → SPL → U-Boot → Kernel → RootFS
- 固件 / 系统分区防篡改
- 防回滚保护（Rollback Protection）
- 分区完整性校验（hash / hashtree）
- 设备锁（Device Lock）
- 基于 RK3568 + U-Boot + OH 6.1 L2 直接落地

## 3. 核心结论

**不必从零实现 OHVB。** OpenHarmony 官方已有完整的 **HVB（Harmony Verified Boot）** 部件，架构与 OHVB 目标一致：

- 验签协议：`libhvb` + RSA/SM2 + SHA256
- 分区尾部签名：HVB Footer + Cert
- 根校验表：**RVT（Root Verity Table）** 分区
- 防回滚：RPMB 存储 rollback_index
- 设备锁：OTP / TEE lock_state
- 内核运行时：dm-verity（init 通过 fstab `hvb` 标志启用）

Kaihong 侧已有扩展：

- `kh_hvbtool.py`：密钥生成、永久属性、unlock 等
- `hvb_sign.py`：构建后批量签名
- `bootkhp.c`：U-Boot 启动流程集成 `hvb_verify()`

**建议**：对外统一使用 **OHVB（OpenHarmony Verified Boot）**；对内实现直接复用并增强 HVB。

## 4. 约束条件

本方案 v1.0 **明确不做**：

1. BIOS 安全启动菜单实现
2. BIOS 内安全策略开关 / 配置项

策略管理通过 **CLI + 产线脚本 + fastboot/rockusb + OP-TEE** 完成，详见 [06-运维与管理.md](./06-运维与管理.md)。

## 5. 技术亮点（对外表述）

1. OpenHarmony 原生 HVB + RK3568 硬件信任根，非 Android AVB 简单移植
2. U-Boot → init → dm-verity 三端闭环，工业级分区防篡改
3. RPMB 防回滚 + OTP 设备锁，满足 ISO 26262 / 等保启动完整性要求
4. 与 OpenHarmony 构建链深度集成（kh_hvbtool、hvb_sign、bootkhp 已有基础）
5. 分阶段 fuse：开发态软校验 → 产线硬锁，降低变砖风险
