# 06 - 运维与管理

## 0. U-Boot BIOS（已实现）

KaihongOS 在 **U-Boot 阶段** 提供 HDMI 图形 BIOS（`kh_bios`），配置目录：

`device/board/kaihong/khdvk_rk3568_a/uboot/kh_bios/`

| 操作 | 入口 |
|------|------|
| 进入 BIOS | 上电倒计时内 **USB F10**，或串口 `kh_bios` |
| 启动配置 | Boot Configuration：Normal / Recovery / 下次 Recovery |
| 硬件参数 | Hardware：GMAC 开关等（`kh_gmac0/1` env） |
| 高级功能 | Fastboot / Rockusb / U-Boot CLI |
| 安全状态 | Security 页：HVB 占位（完整 OHVB 见 Phase 5） |
| 保存并启动 | Save Configuration and Exit → `bootkhp` |

HVB 安全启动策略**不在 BIOS v1 中修改**；待 HVB Phase 0 完成后扩展 Security 页。

## 2. 产线 Provisioning 流程

### 2.1 首次出厂（建议脚本 `provision.sh`）

```
1. 生成 / 导入 PRK、PSK 密钥对
2. 生成 product_id.bin、salt.bin
3. 生成 permanent_attributes.bin
4. 签名 permanent_attributes（secure_boot key）
5. 编译 + hvb_sign.py 签名全部镜像
6. 刷入完整 GPT 镜像
7. 首次启动：U-Boot 写入 permanent attributes 到 OTP
8. 验证 HVB 链通过
9. 设置 lock_state = 0（锁定）
10. 记录设备 SN ↔ 公钥指纹映射
```

### 2.2 密钥文件清单

| 文件 | 存储位置 | 说明 |
|------|----------|------|
| `keys/prk_priv.pem` | 离线 HSM / 保险柜 | 根私钥，永不入设备 |
| `keys/psk_priv.pem` | 签名服务器 | 分区签名私钥 |
| `keys/prk_pub.pem` | 设备 OTP | 根公钥 |
| `product_id.bin` | 设备 OTP | 产品标识 |
| `permanent_attributes.bin` | 设备 OTP | 永久属性 |
| `permanent_attributes_cert.bin` | 设备 OTP | 属性证书 |

## 3. 设备 Lock / Unlock

### 3.1 Lock（出厂默认）

- `rk_hvb_read_lock_state()` 返回 `0`
- `hvb_verify()` 执行完整链式校验
- 校验失败 → 拒绝启动

### 3.2 Unlock（维修 / 研发）

**流程**：

```
PC 工具                    设备
  │                          │
  ├─ 生成 challenge ────────→│
  │                          ├─ OP-TEE 验证
  │←── 返回 response ────────┤
  ├─ 验证 response           │
  ├─ 发送 unlock token ─────→│
  │                          ├─ 写 lock_state=1 到 OTP
  │                          └─ 重启
```

**unlock 后行为**（`uboot_hvb.c` 已有逻辑）：

```c
if (lock_state == 1) {
    printf("[hvb] Warning! unlocked\n");
    return HVB_OK;  // 跳过校验
}
```

**kh_hvbtool 相关命令**（参考 `kh_readme.md`）：

- `make_htx_permanent_attributes`
- unlock challenge / token 相关（Phase 2 完善）

## 4. 状态查看

### 4.1 U-Boot 串口

关键 log 关键字：

```
[hvb] bootloader boot flow start hvb verify
[hvb] Warning! unlocked          ← unlock 状态
[hvb] hvb verify finish in uboot, errorcode: 0x0   ← 成功
[hvb] bootloader boot flow hvb verify failed       ← 失败
```

### 4.2 系统运行时（建议 CLI `ohvb-status`）

可读取的信息：

| 信息 | 来源 |
|------|------|
| HVB 是否启用 | `/proc/cmdline` → `ohos.boot.hvb.enable` |
| dm-verity 状态 | `dmsetup status` |
| lock 状态 | OP-TEE TA 查询 |
| rollback index | RPMB 读取 |
| 公钥指纹 | kh_hvbtool print_sha256 |

### 4.3 示例命令

```bash
# 查看 bootargs 中的 HVB 参数
cat /proc/cmdline | tr ' ' '\n' | grep hvb

# 查看 dm-verity 设备
dmsetup status

# 查看 system 挂载选项
mount | grep -E 'usr|vendor|sys_prod'
```

## 5. OTA 升级与重签名

OTA 升级后分区内容变化，必须重新签名：

```bash
# 1. OTA 刷入新 system/vendor 等
# 2. 重新签名
python3 hvb_sign.py sign --product_name khdvk_rk3568_a

# 3. 刷入更新后的镜像（至少 rvt + 变更分区）
```

**注意**：新版本的 `rollback_index` 必须 ≥ 设备 RPMB 中存储的值，否则启动失败。

建议封装为 `ota_resign.sh` 集成到 OTA 流水线。

## 6. 救砖流程

| 场景 | 方法 |
|------|------|
| HVB 校验失败（lock 态） | MaskROM → rockusb → 刷完整 signed 镜像 |
| 开发调试（unlock 态） | 直接刷 unsigned 镜像 |
| OTP 已 fuse 后 | 只能刷 **对应密钥签名** 的镜像 |
| U-Boot 损坏 | MaskROM → 刷 MiniLoaderAll + uboot.img |

**产线必须保留**：

1. MaskROM 引脚 / 按键操作文档
2. unsigned loader 备份（fuse 前）
3. 对应 PRK 签名的完整救砖镜像

## 8. BIOS 与 HVB 的衔接

U-Boot BIOS 已提供 **Security** 只读页；HVB 启用后将在该页显示：

- HVB 状态、lock 状态、rollback index、公钥指纹
- unlock 需工厂 token（调用 `rk_hvb_ops_user`）

与早期「无 BIOS」方案的关系：日常启动配置走 BIOS；产线 HVB fuse / unlock 仍可走 CLI + `kh_hvbtool`。
